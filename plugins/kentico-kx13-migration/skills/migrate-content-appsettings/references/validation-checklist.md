# Generated Configuration Validation Checklist

Before presenting the generated appsettings.json to the user, verify all of the following. If any check fails, fix the issue and re-validate before proceeding.

1. **Parse check:** Verify the JSON is valid (no trailing commas, no comments, balanced braces/brackets).
2. **Semicolon-separated strings:** Confirm `ConvertClassesToContentHub` and `CreateReusableFieldSchemaForClasses` are semicolon-separated strings, not JSON arrays.
3. **Nesting:** Verify target connection string is under `Settings.XbyKApiSettings.ConnectionStrings.CMSConnectionString`, not at the top level.
4. **Exclusions completeness:** Every class from the plan's Exclusions section appears in `EntityConfigurations.CMS_Class.ExcludeCodeNames`, and no extra classes are present.
5. **Exclusions inheritance safety:** No class in `ExcludeCodeNames` is a parent class (via `ClassInheritsFromClassID`) of any class being migrated. If the plan includes a class inheritance hierarchy section, cross-reference it. If not, warn that inheritance was not validated.
6. **FieldMigrations completeness:** Each entry has all required properties: `SourceDataType`, `TargetDataType`, `SourceFormControl`, `TargetFormComponent`, `Actions`, `FieldNameRegex`.
7. **No empty collections:** No empty arrays `[]` or empty objects `{}` in the output.
8. **EntityConfigurations key naming:** Verify that all keys under `EntityConfigurations` use underscores, not dots (e.g., `"CMS_Class"`, not `"CMS.Class"`). Dots silently break `ExcludeCodeNames` and other entity-level settings.
9. **No TODO/PLACEHOLDER values:** Scan the entire generated JSON for `[TODO]`, `<PLACEHOLDER>`, `TODO:`, or `CHANGEME` values. These cause runtime crashes (`FormatException` for GUIDs, connection failures for strings). If any are present because infrastructure values could not be discovered (Step 1), warn the user explicitly — do not silently emit placeholders.
10. **KxCmsDirPath exists and contains media files:** If `KxCmsDirPath` was set (from Step 1b discovery or the plan), verify the directory exists on disk. If it does not exist, warn that media migration and attachment migration will fail. Additionally, check whether the directory actually contains media library files by looking for a `media/` subfolder (and optionally `files/` for page attachments). If `KxCmsDirPath` exists but has no `media/` subfolder — or the subfolder is empty — **always ask the user** how to proceed. Present the options: (a) keep the current `KxCmsDirPath` as-is (media migration will produce zero assets; user can copy/symlink media folders into the directory before running migration), or (b) change `KxCmsDirPath` to an alternative directory that contains those folders (list any candidates found in the workspace). **Never change `KxCmsDirPath` away from the standard CMS root directory without explicit user confirmation** — even if a directory with media files is found elsewhere in the workspace. The user may have valid reasons to keep the standard CMS root (e.g., planning to copy media files later, or intentionally skipping media migration).
11. **MigrationProtocolPath is absolute and at workspace root:** If `MigrationProtocolPath` is set, verify it is an absolute path pointing to a `MigrationProtocol/` directory under the workspace root (the common parent of KX13, XbyK, and migration tool projects) — not inside any individual project. The migration tool auto-creates the parent directory, so existence checks are not needed — but a relative path may resolve unexpectedly depending on the working directory at runtime.
